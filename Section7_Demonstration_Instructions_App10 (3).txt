BEFORE YOU BEGIN -- SETUP FOR ALL DEMONSTRATIONS
------------------------------------------------------------------------

1.  Connect the Arduino Uno R3 to your PC via USB. Make sure the LED
    is wired to digital pin D3 through a 220 ohm series resistor, and
    that the LDR voltage divider (with 5.1 kohm series resistor) feeds
    analog pin A0.

2.  Verify your COM port:
      Windows  -> Device Manager -> Ports (COM & LPT)
      Mac/Linux -> check /dev/ttyUSB0 or /dev/ttyACM0
    Open app10_advanced.m in MATLAB App Designer and update the line:
        a = arduino('COM5', 'Uno');
    replacing 'COM5' with your actual port before running.

3.  Open the app in MATLAB App Designer and press Run (F5).

4.  The app launches and auto-calibrates immediately:
      - The LED turns ON briefly (measuring SetPointMin -- the lower
        voltage at A0 corresponding to maximum brightness).
      - The LED then turns OFF (measuring SetPointMax -- the higher
        voltage at A0 in darkness).
      - The Actual Value gauge, SetPoint slider, and SetPoint edit
        field limits are all set automatically based on the measured
        voltage range. The exact range will depend on your ambient
        light, the LED, and the geometry of your tissue diffuser, so
        expect values different from any other student's setup.
      - The loop then idles at the midpoint setpoint, waiting for
        you to press START.

5.  Place a small piece of folded tissue between the LED and the LDR
    to diffuse the light evenly. Do not change this physical setup
    between demonstrations.

6.  Set all gain sliders (Kp, Ki, Kd) to ZERO before each new
    demonstration unless otherwise stated below.

7.  The Gains Changing switch controls how slider changes apply:
        OFF (default) -> gain takes effect when you RELEASE the slider
        ON            -> gain updates continuously while dragging
    Use ON mode when you need real-time response during tuning, for
    example when sweeping Kp upward in 7.4.

[APP10 NOTE -- Derivative on the Process Variable]
  In App10 the derivative is computed on the FILTERED process
  variable, not on the error:
      FilteredPV     = 0.8*FilteredPV + 0.2*ActualValue
      DerivativeTerm = (FilteredPV - PreviousFilteredPV) / dt
  The exponential moving average (alpha = 0.2) smooths sensor noise
  before differentiation. The result is in volts per second (V/s),
  not V/sample. This means abrupt setpoint changes do NOT cause a
  derivative kick -- the derivative reacts only to actual changes
  in the measured signal. You will see this difference clearly in
  7.1 when stepping the setpoint.

[APP10 NOTE -- Integral Scaling by dt]
  In App10 the integral is scaled by the real loop time:
      Errorint = Errorint + (Error * dt)
  where dt is the actual elapsed time between iterations. This
  means the integral fills up at a rate that reflects real seconds
  rather than sample count. The anti-windup clamp is set to +/- 50
  V*s (IntErrorMax = 50, IntErrorMin = -50). The ErrorInt gauge
  has matching limits.

[APP10 NOTE -- PID Sign Convention]
  Because the derivative is now on PV instead of error, App10 uses
  a different sign for the Kd term inside the formula:
      OutputVoltage = -(Kp*Error + Ki*Errorint - Kd*DerivativeTerm)
  The Kd term is subtracted inside the brackets so that, after the
  outer negation, it produces a damping action that opposes the
  direction of change in the PV.

[APP10 NOTE -- Recording Buffer]
  Data vectors are pre-allocated to 10,000 samples, which is enough
  for many minutes of continuous recording at the typical loop
  period of this app. The app automatically stops and shows a
  uialert if you reach the 10,000 sample limit. STOP and SAVECSV
  between scenarios so each demo has its own file.

[APP10 NOTE -- True Time Logging]
  App10 logs real elapsed time (TimeVec) per sample, not just sample
  index. This means the Time_s column in your CSV is accurate
  seconds, which makes it easier to plot and discuss recovery times
  in your report.

7.1  TRACKING -- INTEGRAL-ONLY CONTROL (Ki ONLY)
========================================================================

Goal:
  Demonstrate that the controller follows a changing setpoint using
  integral action alone. This is called "tracking." Use both the
  SetPointSlider and the SetPointEditField to vary the setpoint, as
  required by Section 7.1 of the project guidelines.

Gain settings:
  Kp = 0    Ki = 1    Kd = 0

Steps:
  1.  Set Kp = 0, Ki = 1, Kd = 0 using the sliders.
  2.  Press START. The system should hold the calibrated midpoint
      setpoint, and the PWM output will settle at whatever value
      keeps the Actual Value at that setpoint.
  3.  Drag the SetPointSlider to a higher value (somewhere near
      the top end of the calibrated range). Observe the Actual
      Value follow the new setpoint over a few seconds. The PWM
      output drops because a higher target LDR voltage means the
      LED needs to dim.
  4.  Type a different value directly into the SetPointEditField
      (try a value near the bottom of the calibrated range) and
      press Enter. Confirm that the slider updates to match. The
      PWM output rises toward saturation as the controller tries
      to brighten the LED enough to push the LDR voltage down.
  5.  Continue moving the setpoint back and forth so tracking is
      clearly visible on the live plot.
  6.  Press STOP, then SAVECSV.

What to observe:
  - The Actual Value trace follows the SetPoint trace in subplot 1
    of the live plot. After each step change, expect the Actual
    Value to converge onto the new setpoint within a handful of
    samples.
  - The ErrorInt gauge accumulates while error is non-zero, then
    flattens once the error reaches zero. Because integration is
    scaled by dt, this trace moves more gently than it would in
    App7.
  - The PWM output adjusts continuously, dropping toward 0 V on
    the way to a high setpoint and rising toward 5 V on the way
    to a low setpoint.

[APP10 NOTE -- No Derivative Kick]
  When you change the setpoint, watch the Derivative (dPV/dt)
  gauge (the 4th gauge / 4th subplot). In App10 this gauge stays
  calm during setpoint steps because the derivative is computed
  on the FILTERED process variable, not on the error. You will
  only see motion on this gauge when the Actual Value itself
  changes -- not when you move the setpoint slider. Mention this
  clearly in your report as the "derivative on PV" improvement
  over App7.

What to include in the report:
  - Show the output graph segment where Actual Value follows
    SetPoint through at least two step changes.
  - Explain that the integral term accumulates error over time
    until error is driven to zero, enabling perfect setpoint
    tracking.
  - Highlight that the derivative trace remains calm during
    setpoint steps -- this is the "derivative on PV" behaviour
    and contrasts with App7 (where you would see derivative kicks
    on every setpoint change).


========================================================================
7.2  PROPORTIONAL-ONLY CONTROL -- STEADY-STATE ERROR
========================================================================

Goal:
  Demonstrate that proportional action alone cannot eliminate the
  steady-state offset between the setpoint and the actual value.
  Section 7.2 of the project guidelines specifies Kp = 1, Ki = 0.

Gain settings:
  Kp = 1    Ki = 0    Kd = 0

Steps:
  1.  Set Ki = 0, Kd = 0, Kp = 1.
  2.  Press START.
  3.  Move the SetPointSlider to a value well away from the current
      Actual Value. Try several setpoints across the range so the
      offset behaviour is visible at multiple operating points.
  4.  Wait for the system to settle at each setpoint. Observe that
      the Actual Value moves toward the setpoint but stops short
      -- a visible gap remains between the SetPoint and Actual
      Value traces.
  5.  Press STOP and SAVECSV.

What to observe:
  - The Error gauge reads a non-zero value at steady state for any
    setpoint that requires the LED to be partially on. The size of
    the residual error grows as the demanded change from the
    natural rest point grows.
  - The ErrorInt gauge drifts toward its clamp, but the rate is
    slower than in App7. With Ki = 0 this saturated integral has
    no effect on the output, but it is still visible on the gauge
    and plot.
  - The PWM output settles at a constant value proportional to the
    error. Larger errors produce larger constant outputs.

[APP10 NOTE -- Slower Integral Drift Than App7]
  Because the integral is multiplied by dt, the integral fills up
  more slowly than in App7 for the same physical error. This is
  one reason App10's anti-windup limit feels less aggressive in
  normal operation -- the integral simply takes longer to reach
  the clamp.

What to include in the report:
  - Show the output graph where Actual Value approaches but does
    not reach the SetPoint line.
  - Explain that with proportional control only, the output is
    proportional to the error. As the error shrinks, so does the
    corrective action, leaving a residual steady-state error that
    sustains the small output needed to hold the current position.
  - Note that ErrorInt still climbs toward its clamp because the
    integral is computed every iteration regardless of Ki -- this
    just demonstrates that the anti-windup machinery is working.


========================================================================
7.3  INTEGRAL WINDUP -- DEMONSTRATION AND RECOVERY
========================================================================

Goal:
  Demonstrate integral windup by letting the error persist with all
  gains at zero, then show recovery by restoring Ki to a reasonable
  value (0.4) as required by Section 7.3 of the project guidelines.

Gain settings (Phase 1 -- Windup):
  Kp = 0    Ki = 0    Kd = 0

Gain settings (Phase 2 -- Recovery):
  Kp = 0    Ki = 0.4    Kd = 0

Steps:
  PHASE 1 -- Cause windup:
  1.  Set all gains to 0.
  2.  Press START. Because all gains are zero, the PWM output is
      held at 0 V regardless of the error -- the LED stays off and
      the LDR sees only ambient light. Error sits at a large
      negative value continuously (Actual is high near SetPointMax,
      SetPoint is at the midpoint).
  3.  Watch the ErrorInt gauge ramp downward.

  [APP10 NOTE -- Windup Is Slower Than in App7]
    Because the integral in App10 is scaled by dt:
        Errorint = Errorint + (Error * dt)
    each step adds only a small fraction of the error to the
    running total -- much less than App7's per-sample sum. Expect
    the ErrorInt gauge to take noticeably longer to reach its
    clamp of -50 V*s than it does in App7. Be patient -- the
    units on the gauge are V*s (volt-seconds), not plain volts.

  4.  Confirm that the ErrorInt gauge has reached its clamp at
      -50 V*s before moving to Phase 2.

  PHASE 2 -- Demonstrate recovery:
  5.  Set Ki = 0.4 (leave Kp and Kd at 0).
  6.  Observe that the large negative integral drives the PWM
      output high. Expect the output to jump immediately to its
      5 V saturation limit because the (negated) accumulated
      integral is much larger than 5 V.
  7.  The LED turns on at full brightness and the Actual Value
      drops sharply. Expect a deep undershoot below the setpoint
      because the LED is briefly being driven harder than steady
      state requires.
  8.  Once the Error changes sign, the integral begins to unwind.
      The output drops gradually back toward a steady value as the
      ErrorInt gauge climbs back toward zero. The Actual Value
      should eventually settle near the setpoint.
  9.  Press STOP and SAVECSV.

What to observe:
  - During Phase 1, the ErrorInt plot ramps roughly linearly
    downward and flattens at -50 V*s. The ramp is more gentle than
    in App7 because of dt scaling.
  - On restoring Ki, the output saturates at 5 V because the
    saturated integral demands a large LED brightness.
  - The Derivative (dPV/dt) gauge shows a single large negative
    spike at the moment the LED suddenly comes on. This is a real
    derivative response (the PV is genuinely changing rapidly),
    not noise.
  - As the integral unwinds, the output decreases smoothly. The
    Actual Value trace shows a deep undershoot followed by a slow
    recovery -- the classic windup symptom.

What to include in the report:
  - Show the output graph spanning both phases.
  - Explain that windup occurs because error accumulated while the
    output was unable to respond (Ki = 0). Recovery occurs as the
    integral discharges once Ki is restored.
  - State explicitly that the +/- 50 V*s clamp prevents the
    integral from growing unboundedly.
  - Discuss the deep undershoot during recovery as a direct
    consequence of the saturated integral being released suddenly.
  - Note that the dPV/dt gauge spike at the moment of LED turn-on
    is a genuine derivative response to the rapid change in PV --
    this is exactly when derivative damping would help, if Kd were
    enabled.

7.4  INSTABILITY -- EXCESSIVE GAIN DEMONSTRATION
========================================================================

Goal:
  Demonstrate that setting gains too high causes the system to
  oscillate and become unstable, as required by Section 7.4 of the
  project guidelines. Observe instability on the gauges and on the
  physical LED.

Gain settings (start stable, then increase in stages):
  Phase A: Kp = 0.4    Ki = 0.4    Kd = 0.8   (stable starting point)
  Phase B: Kp = 2.0    Ki = 0.4    Kd = 0.8
  Phase C: Kp = 2.0    Ki = 1.0    Kd = 0.8
  Phase D: Kp = 2.0    Ki = 1.0    Kd = 10.0  (unstable)

Steps:
  1.  Begin with stable gains: Kp = 0.4, Ki = 0.4, Kd = 0.8.
  2.  Press START and confirm the system settles at the midpoint
      setpoint with a steady non-zero PWM output.
  3.  Allow the system to settle, then raise Kp to 2.0. The Actual
      Value may briefly dip but should return to the setpoint.
  4.  Once the system has stabilised again, raise Ki to 1.0. You
      may see a small overshoot, but the system should still
      recover.
  5.  Once the system has stabilised again, raise Kd to 10.0. The
      system should break into sustained large-amplitude
      oscillation.
  6.  Observe the LED itself -- it visibly flickers between bright
      and dim as the PWM output swings between its 0 V and 5 V
      saturation limits.
  7.  Press STOP to stop the oscillation, then SAVECSV.

What to observe:
  - Once Kd reaches its maximum, the Actual Value oscillates
    across much of the calibrated range, and the PWM output
    saturates alternately at 0 V and 5 V.
  - The Derivative (dPV/dt) gauge swings widely as it tracks the
    rapidly changing PV.
  - The LED flickers visibly. Do not run this for long --
    saturating the PWM output at high frequency stresses the
    components.

[APP10 NOTE -- High Kd Alone Is Less Likely to Destabilise]
  In App10 the derivative acts on a FILTERED PV (exponential
  moving average, alpha = 0.2), which removes most of the
  high-frequency sensor noise before differentiation. This means
  that in App10, raising Kd alone (with Kp and Ki low) is unlikely
  to cause visible instability -- the filter prevents the
  derivative term from amplifying noise. To force instability you
  generally need to drive Kp and Ki high as well, exactly as in
  the stages above. Compare this with App7, where high Kd alone
  is enough to break the loop. Discussing this contrast in your
  report demonstrates that you understand the role of the PV
  filter.

What to include in the report:
  - Show the output graph segment where oscillation is clearly
    visible across all five subplots.
  - Explain that high Kp amplifies even small errors into large
    corrective outputs, which overshoot and cause the error to
    reverse sign -- producing oscillation.
  - Explain that App10's PV filter (alpha = 0.2 EMA) reduces
    derivative noise sensitivity, which is why pushing Kd alone
    is much less destabilising in App10 than in App7. The
    instability you triggered required the combined effect of
    high Kp, Ki, and Kd.

7.5  EXTERNAL DISTURBANCE REJECTION
========================================================================

Goal:
  Demonstrate that the closed-loop controller compensates for
  external changes in light conditions by covering and uncovering
  the LDR, as required by Section 7.5 of the project guidelines.

Gain settings (stable tuned values):
  Kp = 0.4    Ki = 0.4    Kd = 0.8

Steps:
  1.  Set Kp = 0.4, Ki = 0.4, Kd = 0.8.
  2.  Press START and wait for the system to settle at the setpoint
      (the midpoint of the calibrated range). The PWM output should
      hold at a steady non-zero value.
  3.  DISTURBANCE 1 -- Cover the LDR with your finger or a piece of
      opaque material. Within a sample or two the Actual Value will
      jump upward (the LDR sees less light through the cover). The
      Error becomes strongly negative, and the controller drives
      the PWM output toward its 5 V saturation limit trying to make
      the LED bright enough to compensate. Hold the cover in place
      for roughly half a minute so the disturbance is clearly
      visible on the plot.
  4.  DISTURBANCE 2 -- Remove the cover suddenly. The Actual Value
      will drop downward (the LDR now sees the saturated LED at
      full brightness directly). The error flips sign, the output
      drops, and the system begins recovering toward the setpoint.
      Recovery is gradual and may take tens of seconds because the
      integral term needs to unwind from its clamp.
  5.  Repeat cover/uncover at least once more so the behaviour is
      clearly visible in the output graph.
  6.  Press STOP and SAVECSV.

What to observe:
  - Each disturbance causes a sharp jump on the Actual Value plot,
    followed by a slower recovery toward the SetPoint line.
  - The Error gauge spikes during the disturbance and then returns
    toward zero.
  - The PWM output rises and saturates at 5 V when the LDR is
    covered, and drops sharply when the cover is removed.
  - The Derivative (dPV/dt) gauge shows brief spikes at each
    transition. These are genuine PV derivative responses (not
    noise) and contribute correct damping action through the Kd
    term.
  - The ErrorInt trace rides its clamp during the cover phase and
    unwinds after the cover is removed.

What to include in the report:
  - Show the output graph segment containing at least two
    disturbance events and their recoveries.
  - Explain that the feedback loop continuously measures the
    Actual Value and recalculates the error. When a disturbance
    changes the LDR voltage, the resulting error is detected
    immediately and the PID controller adjusts the LED output to
    restore the setpoint.
  - Note that this behaviour would not be possible with feedforward
    (open-loop) control. This directly answers Question 2 of the
    project guidelines.
  - Mention that during the cover phase the PWM output may
    saturate at 5 V because the LDR voltage rose so far above the
    setpoint that even maximum LED brightness could not bring it
    back. The physical disturbance simply exceeded the controller's
    actuation range -- a hardware limit, not an algorithmic
    failure.
  - Note that the dPV/dt gauge spikes at each cover/uncover event
    are genuine sensor changes, and they contribute correctly to
    the damping action via the Kd term.

  [ ] Describe the sensor measurement:
        The LDR voltage at analog pin A0 is averaged over NrSamples
        readings to reduce noise. The LDR resistance is calculated
        using the voltage divider formula (this is the answer to
        Question 1 of the project guidelines):
            R_LDR = R_Series * (V / (Vcc - V))
        where R_Series = 5100 ohms, Vcc = 5 V, and V = ActualValue.

  [ ] Describe the error calculation:
        Error = SetPoint - ActualValue
        Error can be positive (Actual below SetPoint) or negative
        (Actual above SetPoint).

  [ ] Explain the PID output formula used in App10:
        OutputVoltage   = -(Kp*Error + Ki*Errorint - Kd*DerivativeTerm)
        FilteredPV      = 0.8*FilteredPV + 0.2*ActualValue
        DerivativeTerm  = (FilteredPV - PreviousFilteredPV) / dt
        Errorint        = Errorint + (Error * dt)

      Note the sign of each term:
        - The overall negative sign is required because the system
          is physically inverted (brighter LED -> lower LDR
          voltage).
        - The Kd term appears as MINUS inside the brackets so that
          after the outer negation it provides correct damping
          (opposing the direction of change in the PV).

  [ ] Explain the derivative computation specific to App10:
        The PV is smoothed by an exponential moving average with
        alpha = 0.2 before being differentiated. This avoids
        derivative kick on setpoint changes and reduces noise
        amplification.

  [ ] Explain the integral scaling specific to App10:
        Errorint is scaled by real elapsed time dt rather than
        being summed per sample. This gives a more physically
        meaningful integral in V*s units.

  [ ] Connect the mathematical behaviour visible on the gauges to

  [ ] For 7.3: explicitly discuss the anti-windup clamp (+/- 50
